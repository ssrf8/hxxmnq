## 二、产品定义

玩家继承一座位于结界夹层、会在幻想乡不同地点之间移动的庭园。玩家修复庭园、建设幻想设施，设施形成新的结界锚点并吸引角色来访。角色带来互动、委托与小型异变，事件结果继续改变设施、关系和连接地点。

```text
修复与建设
→ 形成结界锚点
→ 吸引角色与地点连接
→ 进行持续多轮互动和事件
→ 改变庭园、关系与连接
→ 解锁新的建设及移动能力
```

建设是剧情舞台，不是城市经营。第一版不制作生产链、人口、税收、市场、等级、装备、技能树、传统背包或势力战争。

## 三、移动庭园设定

- 庭园主体位于幻想乡边缘的结界夹层，内部空间相对稳定。
- 庭守钥可以识别、开启和稳定连接点。
- 祖父只是上一任持钥者和背景人物，不承担高频主线叙事。
- 庭园早于祖父存在，祖父并非创造者或幕后黑手。
- 祖父曾因两个地点连接重叠造成短期空间错位，出于谨慎关闭全部锚点并返回外界。
- 事故没有宏大伤亡，只留下散落物品、少量未解释记录和连接故障。
- 祖父后来正常去世，庭守钥按安排传给玩家。
- 庭园具有有限自主反应；玩家能通过设施、愿望和庭守钥影响连接，但不能精确输入目的地。
- 主线重点是玩家如何重新经营庭园，而非反复追查祖父。

庭园同时维持：

- 1 个主要稳定连接；
- 最多 2 个临时连接；
- 其他连接处于关闭或等待条件状态。

第一版仅绘制庭园像素地图。前往博丽神社、魔法森林等外部地点时使用正常文字剧情，结束后返回庭园。

## 四、玩家与开场

玩家是祖父后代，从外界进入庭园。玩家不预设成熟超自然能力。

0 号楼开场设置页收集：

- 姓名；
- 称谓或代词；
- 外貌简述；
- 庭园名称。

快速开始优先读取当前 Persona；缺失时使用中性称谓与“无名庭园”。开场草稿不是 MVU 状态，预览不写变量。确认后由本地 bridge 将共享初始状态和已确认资料写入首个 assistant 楼层并复读验证，不调用 LLM；只有验证成功才进入庭院。玩家进入庭院后的第一次真实行动才通过正常 user 消息与生成链开始剧情。

开场顺序：

1. 玩家穿过结界抵达荒废庭园；
2. 短暂自由观察并发现庭守钥；
3. 结界波动引来灵梦；
4. 魔理沙因材料传闻随后出现；
5. 修复主屋并建立时间与休息循环。

卡片不强制叙事人称，也不要求模型输出行动按钮；遵循玩家预设与实际输入。

## 五、庭守钥与能力成长

- 庭守钥用于开启区域、稳定锚点并让玩家参与符卡规则。
- 玩家能力不用等级、经验和攻击力成长表示。
- 只保存具体事实解锁，例如“能看见结界裂隙”“能稳定展开护身结界”“获得基础直射符弹”。
- 每次能力解锁必须来自遗物、角色指导、设施事件或重大剧情。
- 初期能力主要在庭园及其关联异变中生效，避免无依据的强大设定。

## 六、第一版角色名单

固定主要角色共 8 名：

1. 博丽灵梦：结界监督、庭园合法性与封印判断；
2. 雾雨魔理沙：魔法温室、庭守钥实验与自由生长方案；
3. 琪露诺：妖精花园、冻结与局部季节异常；
4. 爱丽丝·玛格特洛依德：温室精细维护、人偶协作方案；
5. 米斯蒂娅·萝蕾拉：夜间活动、宴会饮食与林地来访；
6. 伊吹萃香：宴会、地点重叠、符卡活动；
7. 河城荷取：工程、锚点测量、自动化与故障修复；
8. 十六夜咲夜：宴会维护、精确管理与时间异常。

大妖精不属于第一版固定角色，可在后续作为普通来访角色扩展。

普通状态同时在场 0–4 名角色；宴会、异变和组合事件可临时突破。原作角色默认保留自己的正式居所，长期驻留必须经过持续剧情和玩家明确邀请。

原作设定规则：

- 官方游戏、出版物和漫画中的核心设定优先；
- 二创梗不能未经说明成为既定事实；
- 模糊时间线允许跨作品角色自然登场；
- 角色遵守信息边界，不能凭数据库或模型视角获得全知；
- 长期性格变化必须有可见剧情支撑。

原创常驻 NPC 可以出现，但必须经玩家明确确认后才获得 UID 和长期档案。临时路人、妖精或小妖怪不建长期档案。

## 七、关系与角色档案

不使用好感、信任、亲密等数值。关系只记录具体、可演绎、仍成立的事实。

每条关系事实包含：

- 事实 ID；
- 涉及对象；
- 简短事实；
- 来源事件；
- 成立时间；
- 当前有效状态；
- 最后确认时间。

矛盾变化使旧事实失效并归档，不同时保留互相冲突的当前事实。每名角色最多保留 12 条当前有效关系事实，超出时合并相近事实。

第一版最多保留 16 个非主要角色动态档案。长期未出现、没有未结事件的档案可归入数据库，需要再次登场时恢复。

## 八、资源、时间与天气

资源：

- 物资：范围 0–20，开局 6；
- 灵感：范围 0–10，开局 1；
- 魔法温室基础建设消耗 4 物资与 2 灵感；
- 资源不足时必须提供探索、角色协助或替代材料路线，不形成死局。

时间：

```text
清晨 → 白昼 → 黄昏 → 夜晚 → 次日清晨
```

- 普通交谈不逐条推进时间；
- 建设、探索、共同工作和正式事件通常推进一个时段；
- 睡觉进入次日清晨；
- 四季各 30 天，春季第 1 天清晨开场；
- 普通天气为晴、阴、雨、暴雨、雾、雪；
- 炎热、严寒、魔力雨、局部冻结等使用独立异常天气字段；
- 天气影响来访、地图和事件，但不能长期锁死关键剧情。

## 九、设施与建设

设施状态：

```text
未发现 / 可建设 / 建设中 / 启用 / 损坏 / 异常 / 封印
```

每个设施保存当前形态和已解锁形态。设施不做重复投入资源的数值升级，形态改变来自剧情与玩家选择。

每个建设区域只放置一个主设施，可附带不独立建档的装饰与剧情变化。同一时间只允许一个主要工程处于建设中。损坏、异常和封印均可恢复，失败不会让设施永久消失。

完整版四个设施：

- 魔法温室；
- 妖精花园；
- 月见温泉；
- 宴会广场。

每个设施至少包含正常使用、关系或功能分支、设施异常或组合事件三层内容，但不强制线性推进。

## 十、最小可玩闭环

制作采用分阶段路线，先完成垂直切片，再扩展完整版。

最小闭环顺序：

1. 开场建档与进入庭园；
2. 灵梦、魔理沙调查；
3. 修复主屋；
4. 通过调查异常生长、听取魔理沙设想或研究祖父图纸获得第二点灵感；
5. 清理温室旧地基，推进一个时段；
6. 搭建并稳定基础魔法温室，推进一个时段；
7. 至少一次正常使用；
8. 至少一次持续多轮角色交流；
9. 温室妖花核心事件；
10. 两阶段 JSON 符卡战；
11. 战斗结果改变温室状态并解锁移动锚点线索。

温室后续可解锁：

- 魔理沙关联的自由生长型；
- 爱丽丝关联的人偶维护型；
- 荷取关联的河童自动化型。

首轮不强迫永久选型，玩家可观察后再改造。

## 十一、持续交互会话

点击角色或设施只进入交互，不等于完成一轮内容。一次交互会话允许跨越任意多轮真实 user/assistant 消息。

当前交互会话记录：

- 会话 UID；
- 类型与状态；
- 地点；
- 参与角色；
- 关联设施或事件；
- 开始时间；
- 当前焦点；
- 最近有效消息 ID；
- 当前有效交流轮数（仅在事件明确需要时使用）；
- 约 600 中文字符以内的覆盖式短摘要；
- 是否已经结算。

同一时间只存在一个主要交互会话。切换角色或设施时不能覆盖旧会话，必须继续、邀请新角色加入，或先离场。

“结束当前聊天”按钮生成一条可编辑的告别或离场文本，作为真实 user 消息发送，由 LLM 完成自然收尾和 MVU 结算。“仅退出界面”只用于恢复显示，不改变剧情。

会话中的普通回复不逐条推进时间，结束时根据实际内容统一判断。刷新、切页面和重新打开聊天时从 MVU 恢复会话。Swipe、删除、停止生成或失败回复不能留下正式结算。

### GAL 单壳交互

玩家正常游玩时始终只看见同一个常驻游戏壳。真实 user/assistant 楼层继续承担上下文、Swipe、重新生成、MVU 楼层状态和聊天导出，但在游戏模式下仅做 DOM 视觉隐藏，不设置 `is_hidden`，也不删除消息。

点击角色、区域或设施后，由本地操作注册表结合 MVU 状态、事件前置与阻断条件生成目标附近的可用选项。LLM 只负责演绎玩家已经选择的行动，不得临时创造可执行按钮、资源消耗或绕过前置条件。

角色互动切换到 GAL 近景页面。一次完整 assistant 回复最多拆分为 6 个旁白、动作或台词片段；玩家点击逐段推进，全部播放完后出现 2–4 个建议回应、手动输入和结束聊天。建议回应和手动输入都创建真实 user 楼层。第一版等待完整回复与 MVU 更新落盘后再播放，不做流式 GAL 解析。

“摸摸头”、送礼、邀请等直接动作进入同一套持续会话，角色是否接受由人物性格、关系事实和当前剧情决定。已有主要会话时点击其他目标，必须选择返回、结束当前会话、在条件允许时邀请加入或取消，不得覆盖旧会话。

结束聊天必须发送一次真实离场消息，由 LLM 完成自然收尾与幂等结算后返回庭院。结束代表完成一次交互回合，但不必然推进时段；简短闲聊通常不推进，长谈、调查、建设和正式事件按实际内容或事件配置推进。

## 十二、事件调度

容量：

- 1 个正在进行的主要事件；
- 最多 3 个等待事件；
- 最多 8 个近期结果摘要；
- 关键结果使用简短永久标记；
- 日常事件只保存冷却与最近一次结果。

每个正式事件必须声明前置条件、阻断条件、参与角色、关联设施、优先级、是否一次性、冷却、期限与允许结果。

模型可以即兴创造不改变长期状态的小型日常插曲。若要产生常驻角色、永久关系事实、设施形态或主线影响，必须升级为带 UID 的正式事件。

等待事件已满时不生成新的低优先事件。事件到期必须产生错过、延期或条件变化等可解释结果，不能静默消失。

## 十三、MVU 状态原则

`stat_data` 是正式游戏状态源。变量保存当前状态，不保存全文流水账。

核心分区：

```text
stat_data
├─ meta
├─ environment
├─ player
├─ garden
├─ resources
├─ areas
├─ facilities
├─ characters
├─ presence_snapshot
├─ interaction
├─ events
├─ anchors
├─ battle
├─ key_items
├─ abilities
├─ memory
└─ uid_counters
```

规则：

- 固定角色、设施与事件使用预登记稳定 ID；
- 动态实体创建唯一 UID；
- 改名只修改显示字段，不改变引用 ID；
- 在场、位置与动作属于每轮覆盖的展示快照；
- schema、初始化、更新规则、模型投影、UI 路径、清理和迁移必须一致；
- 初始化方式以目标 MVU 实际协议为准，使用已验证的禁用 `[InitVar]` 条目作为公共初始数据来源；
- 未知字段行为由 schema 明确定义，不能无提示丢失正式数据；
- 迁移前保留旧状态快照，迁移必须幂等；
- 迁移失败时显示诊断，绝不能静默重置存档。

每轮模型只看到当前时间天气、玩家位置、当前交互或事件、在场角色、相关设施、必要资源和相关关系事实，不发送完整数据库或角色库。

## 十四、数据库增强

MVU 是唯一正式状态源，数据库只做长期归档与检索，不能反向覆盖 MVU。

数据库表：

- 角色经历；
- 事件档案；
- 设施沿革；
- 主线线索。

归档只在 assistant 回复完整、MVU 更新成功且会话正式结束后执行。使用聊天标识、消息 ID、实体 UID 和结算 ID保证幂等。

数据库离线或禁用时核心玩法继续。设置页只显示状态与帮助，不自动启用全局数据库脚本。

每轮最多检索 3 条高度相关摘要，总量约 1200 中文字符。不得自动调用数据库插件的 `callAI`、切换预设、批量改写世界书或导入模板。

## 十五、同层应用

0 号楼承载长期同层应用：

- 庭园页：地图、设施、来访角色和环境；
- 剧情页：真实消息镜像、输入与生成状态；
- 设置/恢复页：显示偏好、诊断、原生聊天恢复和数据库状态。

这里的“0 号楼”是玩家视觉上的唯一游戏壳，不要求技术实现绑定真实消息 0 楼。常驻壳由宿主脚本插入聊天容器，避免长聊天懒加载、Swipe 或消息重建导致应用消失。

真实消息和 MVU 是事实源，同层应用只镜像，不创造第二套剧情状态。原生消息只做视觉折叠，禁止使用 `is_hidden` 改变模型上下文。

最小阶段支持：

- 查看真实消息；
- 输入与发送；
- 停止生成；
- 重新生成；
- 最新 assistant 楼层 Swipe。

编辑旧消息、删除和完整分支回看先调用原生聊天。高级支持在闭环通过后补充。

必须保留“显示原生聊天”和“重新加载同层应用”。桥接失去心跳、初始化超时或出现未捕获错误时自动展开原生聊天。

UI 使用 TypeScript；普通界面使用语义化 DOM，庭园地图和符卡战使用 Canvas 2D。远程界面资源固定版本并经过协议校验，高权限酒馆 API 只由本地 bridge 调用。

## 十六、庭园地图表现

采用俯视角 32×32 网格像素地图，固定庭园底图加逐步解锁区域，支持拖动与有限缩放。窄屏调整镜头，不制作第二套移动端地图。

地图显示：

- 时段、季节、普通天气与异常天气；
- 设施状态和形态；
- 未开放区域；
- 当前来访角色；
- 角色进入、移动、停留、互动和离开；
- 锚点入口与结界效果。

重要约束：

- 玩家小人不出现在庭园地图中；
- 玩家位置仍保存在 MVU，用于区域、交互和剧情判断；
- 点击区域或设施表示玩家前往并交互，不播放玩家寻路；
- 只有来访角色拥有庭园地图行走和待机动画；
- 角色前往幻想乡其他地点的路程不在地图模拟。

## 十七、JSON 符卡游戏

首版即制作真正可玩的简化 JSON 符卡游戏，不使用结果模拟器。

安全协议：

- LLM 只能选择预登记 `config_id` 和有限白名单参数；
- JSON 禁止 JavaScript、HTML、任意 URL 和动态表达式；
- 所有速度、数量、尺寸、伤害和持续时间做范围校验；
- 战斗结算使用唯一 ID，只允许一次。

玩法：

- 玩家自机只在符卡游戏中出现；
- 自动向前方持续射击；
- 普通移动使用较宽弹道；
- 专注模式减速、显示判定点并集中火力；
- 键盘、触控和叙事替代结算均可用；
- 3 条生命，碰撞扣除一命并短暂无敌；
- 首版不加入炸弹和复杂角色技能；
- 每场 1–3 个阶段；
- 支持击破、生存计时和收集目标；
- 结果包括配置 ID、胜负、剩余生命、擦弹、持续时间、命中、伤害、阶段击破、目标完成度和结算 ID。

教学战对象为“温室妖花核心”，不建立常驻角色档案：

- 第一阶段：固定和扇形弹幕；
- 第二阶段：追踪弹和局部安全区；
- 魔理沙提供剧情支援，但不替玩家自动取胜。

结果分为漂亮取胜、勉强取胜、落败和叙事替代结算。所有结果都继续剧情，不造成永久伤残或强制结束。

## 十八、美术与动画规范

参考风格为用户提供的高分辨率四方向 Q 版像素角色图。所有生成素材必须重新生成或绘制，不直接拆取原作游戏资源。

角色规范：

- 统一头身比、像素密度、光源和四方向构图；
- 透明或可安全抠图背景；
- 默认闭嘴、中性或轻微微笑；
- 不允许默认张嘴；
- 只有明确的说话、惊讶或受击表情帧才张嘴；
- 帽子、翅膀、扫帚、锁链、背包等配件轮廓必须分离清楚；
- 第一阶段先生成四方向静态参考，再补三帧行走、呼吸、眨眼和专属待机；
- 左右方向允许在不破坏非对称配件时镜像。

玩家不制作庭园地图四方向小人，只制作符卡自机及战斗特效。

首批素材优先级：

1. 八名主要角色四方向参考；
2. 玩家庭守钥符卡自机；
3. 温室妖花核心两阶段；
4. 春季庭园基础底图；
5. 主屋与魔法温室状态层；
6. 古树、封井与锚点入口；
7. 弹幕、命中与阶段击破特效；
8. 晴、雨、雾和昼夜覆盖层；
9. 核心 UI 图标。

素材缺失时使用占位图或关闭动画，不得导致整页白屏。

GAL 近景立绘采用独立素材注册表。模型只输出受限的 `speaker_id`、反应与姿态标签，界面据此解析本地素材；不得执行模型给出的文件路径或 URL。缺少匹配素材时回退到该角色默认图或通用占位图。首个垂直切片可只使用占位近景图，待消息、结算、Swipe 与恢复链稳定后再批量生成正式反应图片。
