## 一、项目定义

本项目是一张以《东方Project》幻想乡为背景的故事型领地建设 MVU 角色卡。

玩家继承祖父留下的荒废庭园，在修复土地、建设幻想设施的过程中，逐渐吸引幻想乡原作角色来访。设施改变角色的活动方式，角色带来事件、委托和小型异变，事件结果又继续改变庭园。

核心表达为：

```text
建设新的场所
→ 吸引新的角色
→ 产生新的互动与事件
→ 改变角色关系和庭园状态
→ 解锁新的建设与祖父主线
```

领地建设是剧情舞台，不是传统城市经营系统。项目不追求复杂生产链、经济循环、人口管理、势力政治或大规模领土扩张。

## 二、第一版产品目标

第一版需要验证一条完整而稳定的核心循环：

```text
同层像素庭园展示当前状态
→ 玩家点击角色、设施或功能入口
→ 前端形成可读且可编辑的玩家行动
→ 行动作为真实 user 消息发送给 LLM
→ LLM 推进一个聚焦的剧情内容
→ MVU 更新设施、角色、事件、时间等状态
→ 同层前端读取新状态并更新地图
→ 必要时进入符卡小游戏
→ 战斗结果回传并继续剧情
```

第一版的成功标准不是内容数量，而是验证以下闭环可以持续运行：

1. 真实聊天、MVU 状态和同层显示保持一致。
2. 玩家能从地图直接发起角色互动、设施建设和休息行动。
3. 设施变化能在地图上表现，并影响角色来访和剧情触发。
4. 角色当前位置、目标和动作能以像素小人表现。
5. 一次小型异变与符卡战可以从剧情进入、返回并产生后续影响。
6. 长聊天、刷新、切换聊天和生成失败时不会破坏存档或隐藏真实消息。

## 三、已经确认的设计原则

### 3.1 系统范围

第一版保留：

- 幻想乡世界观与庭园背景；
- 玩家身份与祖父主线；
- 领地区域和设施建设；
- 少量物资与灵感资源；
- 原作角色档案与关系状态；
- 角色来访、当前位置和地图动作；
- 时间、季节与天气；
- 日常事件、小型异变和长期主线；
- 简化符卡战；
- MVU 状态系统；
- 同层像素庭园、聊天镜像与操作界面。

第一版不做：

- 传统背包与大量普通物品；
- 装备、等级、技能树和完整 RPG 战斗；
- 复杂任务列表与角色每日行动模拟；
- 生产链、工资、税收、人口和市场；
- 势力外交、政治和领土战争；
- 完整幻想乡地图；
- 高精度寻路和复杂角色 AI；
- 无限生成建筑或数百项长期变量。

少数关键剧情物品可以被记录，例如祖父日记、奇怪钥匙、封井符纸和重要遗物。

### 3.2 内容尺度

- 内容不设置强制尺度限制。
- 第一版不制作专门的恋爱、好感或 NSFW 玩法系统。
- 与原作角色发展恋爱关系被保留为后期可选更新方向，不能在第一版变量结构中抢占核心设计。

### 3.3 玩家身份与原作边界

- 玩家固定为祖父的后代，从外界回到幻想乡庭园。
- 姓名、性别、外貌与个人表达允许自由设定。
- 玩家初期不预设成熟的超自然能力，可在后续剧情中逐渐获得参与符卡规则的方式。
- 东方原作角色以原作核心设定和辨识度为基础，允许为庭园长期生活与互动进行合理二创。
- 故事使用模糊时间线，使不同作品时期的角色能够自然登场。

## 四、开场与长期主线

### 4.1 开场前提

玩家幼年时曾短暂与祖父生活在偏僻庭园。多年后，玩家收到祖父去世的消息和一封迟到的遗书：

> “外面的世界已经没有地方留给它们了。回来吧，把那座庭园重新打开。”

玩家按照遗书指引回到旧居，却穿过结界抵达幻想乡边缘。庭园已经荒废，初始可见内容包括：

- 漏雨的旧屋；
- 长满杂草的中央庭院；
- 生长异常的古树；
- 被符纸封住的井；
- 被雾气或结界遮挡的区域；
- 祖父留下的旧日记和奇怪钥匙。

灵梦因结界波动前来调查，魔理沙因稀有材料的传闻随后出现。两人的到来完成世界观引导，也构成玩家与幻想乡角色的第一次正式接触。

### 4.2 主线谜团

长期主线围绕以下问题逐步展开：

- 祖父为什么能够在幻想乡生活；
- 庭园过去为什么被关闭；
- 古树、封井与结界之间有什么关系；
- 哪些角色认识祖父，她们各自知道什么；
- 庭园是否曾经收容被外界遗忘的存在；
- 祖父是否主动抹除了部分历史；
- 庭园重新开放是否会唤醒一场旧异变。

主线不应每轮推进。默认节奏以建设、日常互动和小型事件为主，只有在修复关键设施、开放区域或完成角色事件后才揭示新的主线线索。

## 五、同层前端总体方案

### 5.1 定位

第一版首先实现“同层前端”，不是普通的消息状态栏。

完整游玩界面由 0 号消息楼层承载。玩家通常停留在这一楼，通过应用内页面完成地图查看、剧情阅读、输入、选项、设施操作和设置。SillyTavern 后台仍持续产生真实 user/assistant 消息，保留提示词上下文、Swipe、分支和聊天存档。

### 5.2 三层结构

```text
真实消息与 MVU 数据层
  负责聊天上下文、Swipe、存档和 stat_data
            ↓
常驻桥接服务层
  负责事件汇聚、消息读取、生成状态、失败恢复和显示模式
            ↓
0 号楼同层应用
  负责像素地图、聊天镜像、输入、弹窗、设置和小游戏入口
```

同层应用只能镜像真实消息和 MVU 状态，不能创建第二套剧情真相。

### 5.3 页面结构

顶部提供稳定的页面切换入口，至少包含：

- 庭园：默认页面，显示像素地图、设施和当前在场角色；
- 剧情：显示真实聊天记录的同层镜像、生成状态、输入框和可用操作；
- 设置/恢复：控制音量、动效、显示偏好、诊断和恢复原生聊天。

地图页与剧情页属于同一个 0 号楼应用内部路由，切换页面不会修改真实聊天或游戏变量。

### 5.4 真实聊天与渲染

- 原生聊天楼层默认只做视觉折叠，绝不能使用 `is_hidden`，因为该字段会改变发送给 LLM 的上下文。
- 聊天页读取并镜像真实消息，保留酒馆助手的显示渲染能力。
- 每个镜像节点保留真实 `message_id`、role 和 Swipe 信息。
- 同层界面未加载完成、失去心跳或发生错误时，必须自动恢复原生消息显示。
- 设置页必须提供玩家可操作的“显示原生聊天”逃生按钮。
- 不删除、不清空、不覆盖真实消息来实现界面效果。

### 5.5 同层动作发送协议

所有地图交互最终都形成玩家能看懂的真实 user 消息，并触发正常 LLM 生成：

```text
点击对象
→ 打开对应操作弹窗
→ 生成行动文本
→ 玩家可以修改或补充
→ 玩家确认发送
→ 创建真实 user 消息
→ 触发正常生成
→ 读取真实 assistant 回复与 MVU 更新
```

已经确认：

- 点击角色不会立即消耗一次生成，而是先打开角色交互窗。
- 点击设施先显示设施信息和可用操作。
- 睡觉等重要行动需要二次确认。
- 生成期间锁定重复提交，避免双击产生两轮相同行动。
- 前端普通操作不直接写入正式 `stat_data`，状态改变由正常剧情与 MVU 更新管线完成。

### 5.6 角色交互弹窗

建议的第一版操作项：

- 交谈；
- 询问近况；
- 邀请帮忙；
- 与当前设施有关的操作；
- 自定义行动。

具体按钮会根据角色、位置、当前事件和设施状态动态显示。最终按钮文案与选项输出规范仍待后续确认。

### 5.7 设施交互弹窗

建议的第一版操作项：

- 查看设施；
- 使用设施；
- 清理、修复或建设；
- 调查异常；
- 邀请在场角色参与；
- 自定义行动。

不可用操作需要显示原因，不能只把按钮静默隐藏。

## 六、像素庭园地图

### 6.1 第一阶段区域

- 主屋；
- 中央庭院；
- 古树区域；
- 封井区域；
- 林地；
- 池塘；
- 坡地；
- 数块被结界遮挡的未知区域。

隐藏区域可用于旧仓库、废弃结界场、外界夹缝石门、妖怪休息所、地下房间和被抹除的庭园角落。

### 6.2 地图表现

地图需要表现：

- 当前时段、季节与天气；
- 已建设、建设中、损坏或异常的设施；
- 尚未开放的区域；
- 当前在场的原作角色；
- 角色前往设施、停留、互动或离开的过程；
- 设施因剧情出现的冻结、开花、损坏、封印或异变效果；
- 战斗发生时进入符卡界面，结束后返回庭园。

### 6.3 角色活动驱动

已经确认采用混合驱动：

- LLM/MVU 决定角色是否在场、当前目标地点、目标设施和特殊动作。
- 前端负责寻路表现、随机闲逛、待机动画和到达后的动作循环。
- 没有剧情目标时，角色可以在允许区域内自主散步。
- 前端自主行为只负责表现，不创造剧情、不消耗资源，也不直接修改正式游戏变量。

### 6.4 剧情聚焦规则

- 点击一名角色后，默认聚焦于玩家与该角色。
- 其他在场角色只有在合理靠近、事件要求或组合剧情条件满足时才能加入。
- 当前存在主要事件时，界面提示玩家继续事件或暂时离开，避免多个事件争夺同一轮。
- LLM 每轮集中推进一个主要内容，不同时铺开大量互不相关的角色和事件。

## 七、时间、季节与休息

已经确认采用四时段：

```text
清晨 → 白昼 → 黄昏 → 夜晚 → 次日清晨
```

基本规则：

- 普通交谈不必强制推进时间；
- 建设、探索和重要事件完成后通常推进一个时段；
- 点击主屋中的睡觉功能，确认后进入次日清晨；
- 季节根据经过天数缓慢变化；
- 天气受季节、剧情与异变共同影响；
- 修改时间时必须同步检查天气、角色在场状态、事件期限和日常事件冷却。

精确的季节长度、天气枚举和时间推进例外仍待确认。

## 八、设施与建设内容

### 8.1 第一版可建设设施

1. 妖精花园
   - 主要关联：琪露诺、大妖精；后续可扩展三月精和莉莉白。
   - 事件方向：冻结喷泉、局部季节错乱、妖精战争、植物说话。

2. 魔法温室
   - 主要关联：魔理沙、爱丽丝；后续可扩展帕秋莉和梅蒂欣。
   - 事件方向：蘑菇繁殖、人偶照料、实验爆炸、有毒植物、魔法生命。

3. 月见温泉
   - 第一版可由八名主要角色参与相关事件；后续扩展永远亭、守矢、红魔馆和地底角色。
   - 事件方向：月光药泉、冻结、古老水脉、温度失控。

4. 宴会广场
   - 主要关联：灵梦、萃香、米斯蒂娅；其他在场角色可形成组合事件。
   - 事件方向：宴会、酒量比赛、符卡表演、未受邀请的来客。

### 8.2 后续设施储备

- 夜间食堂；
- 外界物陈列室；
- 结界符卡场；
- 安静书斋。

这些设施保留在总体设计中，但不要求进入第一版核心闭环。

### 8.3 建设原则

- 建设主要改变可发生的故事，不追求产量升级。
- 设施分支由角色事件和玩家选择产生，而不是重复投入资源升级。
- 同一设施只启用一种当前形态，但可以记录多个已解锁形态。
- 是否允许通过后续剧情反复改造、具体建设阶段和损坏等级，目前仍是待确认提案。

## 九、角色内容

### 9.1 第一版主要角色

- 博丽灵梦；
- 雾雨魔理沙；
- 琪露诺；
- 大妖精；
- 爱丽丝·玛格特洛依德；
- 米斯蒂娅·萝蕾拉；
- 伊吹萃香；
- 河城荷取。

### 9.2 关系表达原则

不使用传统好感数字，优先记录具体而可演绎的当前关系状态，例如：

- 灵梦暂时认可玩家管理庭园；
- 灵梦会定期检查封井；
- 魔理沙在温室拥有实验角；
- 魔理沙私自复制了仓库钥匙。

关系变量只保存当前仍然成立的标签、约定和长期影响，不保存完整互动流水账。

### 9.3 角色档案边界

当前提案为：八名主要角色使用固定档案；其他原作角色只有在产生实质互动且确认会再次登场时才建立动态 UID 档案；一次性无名角色不建档。

第一版是否允许 LLM 创建原创常驻 NPC 尚未确认。

## 十、事件与剧情控制

### 10.1 事件类型

建议分为：

- 开场事件；
- 设施建设事件；
- 角色个人事件；
- 多角色组合事件；
- 可重复日常事件；
- 小型异变；
- 祖父主线事件；
- 符卡挑战事件。

### 10.2 状态要求

事件至少需要表达：

- 尚未触发；
- 等待处理；
- 正在进行；
- 暂缓；
- 已完成；
- 永久关闭；
- 事件结果；
- 对角色、设施和主线产生的长期影响。

### 10.3 调度原则

- 同一时间只推进一个主要事件。
- 触发条件必须读取设施、角色、时间和前置事件状态。
- 已完成的一次性剧情不能重复触发。
- 日常模板允许重复，但需要冷却与结果变化。
- 普通日常结束后进入摘要，不长期占用事件变量。
- 具体的“一个进行中、三个等待”等容量上限尚未确认。

## 十一、资源与关键物品

第一版只保留：

- 物资：普通木料、工具、食材和修缮材料的抽象总量；
- 灵感：从探索、角色互动和异变中获得的特殊建设想法；
- 少量关键剧情物品。

资源不足不应形成死局，应当允许通过探索、请求角色帮助、替代材料或不同剧情路线继续推进。

资源精确范围、开局数值与设施消耗尚未确认。此前提出的 `物资 0～20、灵感 0～10` 仅为讨论提案，不是定案。

## 十二、符卡小游戏

### 12.1 进入条件

符卡战只在剧情自然发展到以下情况时触发：

- 玩家接受角色挑战；
- 角色之间的争执需要用符卡规则解决；
- 设施被异常力量占据；
- 小型异变需要处理；
- 举办符卡比赛；
- 灵梦测试玩家管理庭园的资格。

### 12.2 战前剧情协议

LLM 负责生成或确定：

- 对战原因；
- 对手；
- 战前对话；
- 战斗地点；
- 符卡主题；
- 战斗目标；
- 对应的预设战斗配置标识。

前端不能接受 LLM 任意生成可执行代码；战斗只能从经过校验的预设 JSON 配置和有限参数中选择。

### 12.3 第一版弹幕类型

- 环形弹幕；
- 扇形弹幕；
- 追踪弹；
- 交叉弹幕；
- 随机弹幕；
- 固定安全区域；
- 生存计时；
- 收集目标。

### 12.4 战斗返回协议

战斗结束后向正常剧情回合提交结构化结果摘要，例如胜负、剩余生命、擦弹数和持续时间。LLM 根据结果继续剧情。

战败不能导致游戏结束，可以产生被取笑、设施暂时被占用、替代解决路线、下次再战或新的日常剧情。

## 十三、MVU 状态总体边界

第一版采用 MVU，并推荐 RFC 6902 标准 JSONPatch 方言。正式字段名、类型和路径必须等酒馆制卡 skill 更新后再次确认。

当前概念变量树如下，仅用于规划：

```text
stat_data
├─ 元信息与协议版本
├─ 环境
│  ├─ 日期与经过天数
│  ├─ 当前时段
│  ├─ 季节
│  └─ 天气与异变天气
├─ 玩家与庭园
│  ├─ 玩家基础设定
│  ├─ 庭园名称
│  ├─ 当前建设阶段
│  └─ 当前所在区域
├─ 资源
│  ├─ 物资
│  └─ 灵感
├─ 区域
├─ 设施
├─ 角色档案
│  ├─ 固定主要角色
│  └─ 动态来访角色
├─ 当前在场与地图表现快照
├─ 事件
│  ├─ 当前主要事件
│  ├─ 等待事件
│  ├─ 已完成关键事件
│  └─ 日常冷却
├─ 祖父主线标记
├─ 当前战斗
├─ 关键剧情物品
├─ 摘要与长期备忘
└─ UID 计数器
```

设计约束：

- 变量只保存当前状态，不保存正文流水账。
- 设施、角色和事件的动态实体必须使用稳定 UID，不以显示名称作为动态 key。
- 当前在场、当前位置和当前动作属于快照，每轮覆盖，不持续追加。
- 已完成事件只保存足以影响后文的短结果。
- schema 是唯一事实源；后续所有世界书、EJS、前端与脚本都必须跟随 schema 同步。
- `[InitVar]` 条目必须禁用，由 MVU 标准初始化协议读取。
- 同层前端普通操作不能绕过 MVU 管线直接写正式游戏状态。

## 十四、轻量模式与长期记录

已经确认总体目标需要支持：

- 轻量模式：只依赖卡片、真实聊天和 MVU 变量即可完成全部核心玩法；
- 增强模式：可使用额外数据库保存更长的历史记录，但不能增加轻量模式不存在的玩法。

当前资料将“数据库兼容同层前端”标记为需要真机验证的实验路线，因此第一版生产闭环必须以真实消息 + MVU 为唯一稳定后端。数据库增强只能作为后续适配器研究，不能与第一版稳定数据源混写，也不能宣称已经兼容。

## 十五、第一版内容量

### 15.1 必须包含

- 玩家继承并进入庭园；
- 灵梦与魔理沙前来调查；
- 修复主屋；
- 开放第一个建设区域；
- 四个可建设设施；
- 八名主要角色；
- 每个设施至少三段相关剧情；
- 至少两段多角色组合剧情；
- 一次小型异变；
- 两场符卡战；
- 胜利和战败都能继续故事；
- 揭露一段祖父过去的关键线索。

### 15.2 建议的第一版剧情骨架

```text
序章：归来的继承人
→ 灵梦与魔理沙调查结界波动
→ 修复主屋，建立基本时间与休息循环
→ 清理第一个区域
→ 在四个设施方向中选择首建设施
→ 对应角色开始稳定来访
→ 设施个人事件与组合事件展开
→ 一次设施异常演变为小型异变
→ 触发第一场教学型符卡战
→ 异变结果改变设施形态或区域状态
→ 解锁祖父日记中的一段真实记录
→ 第二场符卡战验证分支结果与失败继续机制
```

- 确认 RFC 6902 方言；
- 确认变量字段、类型、枚举、范围和动态开放边界；
- 编写 schema 与 InitVar；

### 阶段 2：叙事与世界书

- 编写开场、建档协议、演绎指南和文风指引；
- 建立角色、地点、设施、事件与祖父主线条目；
- 设计选择栏和摘要机制；
- 控制常驻 token，按关键词和事件拆分详则。

- 建立 0 号楼 shell、常驻 bridge 和版本化通信协议；

- 实现真实消息镜像、输入、发送、生成状态和失败恢复；
- 实现只影响视觉的原生聊天折叠；
- 实现 Swipe、刷新、切聊天和长聊天恢复的最低支持。

### 阶段 4：像素地图与设施交互

- 绘制或装配第一版庭园地图；
- 实现设施热点、角色像素小人、路径与动作；
- 实现角色和设施交互弹窗；
- 建立 MVU snapshot 到地图表现的单向映射。

### 阶段 5：第一版剧情内容

- 完成序章、四设施事件、组合剧情、小型异变和祖父线索；
- 验证事件不会重复、抢占或产生幽灵角色；
- 验证胜负、拒绝、暂缓和替代建设路线。

### 阶段 6：符卡小游戏

- 建立有限且可校验的战斗 JSON 协议；
- 实现基础弹幕、结算与返回剧情；
- 完成两场第一版战斗和胜败分支。

### 阶段 7：验证与部件交付

- 进行 schema、世界书、脚本、正则和前端同步检查；
- 进行 XSS、不可信文本和远程资源检查；
- 进行双击、停止生成、Swipe、删改消息、刷新、切聊天、断网和长聊天测试；
- 进行桌面、窄屏、触控和软键盘测试；
- 按 skill 要求交付带版本号的可导入部件；
- 只有用户明确要求组装或打包角色卡时，才生成完整角色卡 JSON。
