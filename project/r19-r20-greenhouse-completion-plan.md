# 0.2.0-r19 / r20 魔法温室最小闭环联合实施计划

## 1. 状态与决策

- 计划状态：2026-07-24 已由所有者授权实施；r19+r20 联合候选已完成开发、打包、导入与运行冒烟，现进入人工验收修复。
- 已验收基线：`0.2.0-r18`。
- 开发方式：r19 与 r20 作为一个连续开发批次实施，中间不等待所有者人工验收。
- 交付方式：保留 r19、r20 两个可追踪源码里程碑；r20 集成完成后再统一进入分批真机验收与修复。
- 打包边界：所有者已在实施指令中明确授权打包与导入；最终只生成 `0.2.0-r20` 联合候选，未用含 r20 逻辑的源码伪造 r19 包，已有检查点目录未覆盖。
- 当前目标：完成 M1 最小可玩闭环的后半段，而不是提前展开 M2 四设施、八角色全量内容。

## 2. 基线与目标闭环

r18 已提供：

- 确定性零生成开场；
- 同层常驻游戏壳；
- 灵梦 GAL 对话与持续会话；
- 旧主屋检查／维修样板；
- 真实 user/assistant 消息事务；
- 长正文分页、建议回复、结束聊天与时段别名；
- 两阶段 JSON 符卡战引擎与温室妖花配置骨架。

r19+r20 完成后，玩家应能连续经历：

```text
灵梦检查与主屋修复
→ 魔理沙循魔力痕迹来访
→ 获得第二点灵感
→ 清理温室旧地基
→ 搭建并稳定基础魔法温室
→ 第一次使用温室
→ 与魔理沙进行持续多轮交流
→ 触发温室妖花核心
→ 完成两阶段符卡战或叙事替代
→ 温室状态随结果改变
→ 获得移动锚点线索
```

## 3. 联合开发策略

### 3.1 r19：温室建设与首次使用

r19 负责将现有事件配置接入受控 UI 操作与 MVU 结算，完成：

1. `marisa_material_rumor`：魔理沙来访与温室旧地基解锁；
2. `gain_second_inspiration`：三种叙事入口共享一次性第二点灵感结算；
3. `clear_greenhouse_foundation`：清理旧地基并推进一个时段；
4. `build_basic_magic_greenhouse`：预览并结算 4 物资、2 灵感与一个时段；
5. `greenhouse_first_use`：第一次正常使用与温室状态确认；
6. 温室区域、设施菜单、禁用原因、进行中状态和结果反馈；
7. r18 开场、灵梦、主屋、GAL 正文和原生恢复回归。

r19 是源码里程碑，不作为等待人工验收的停点。完成自动门禁后直接进入 r20。

### 3.2 r20：持续交流、妖花事件、战斗与锚点线索

r20 在 r19 状态链上继续完成：

1. `greenhouse_multiturn_conversation`：至少两轮有效玩家回应后的持续交流结算；
2. `greenhouse_flower_core`：满足前置后进入正式主要事件；
3. 只有该事件活跃且配置 ID 匹配时才能进入妖花战；
4. 战斗 bridge 校验并暂存唯一 `battle.current`；
5. 真实 user/assistant 消息完成战后叙事、幂等结算和状态清理；
6. 按 `clean_win`、`narrow_win`、`loss`、`narrative` 改变温室状态；
7. 写入可供后续 M2 消费的移动锚点线索，但不提前建立正式稳定锚点；
8. r19 全链及 r18 核心能力回归。

r20 完成自动门禁后形成联合候选。之后按验收批次慢慢验证，发现问题使用后续新检查点修复，不回写已交付目录。

## 4. 系统契约

| 系统 | 唯一事实源 | 写入者 | 读取／渲染者 | 结算与恢复 |
|---|---|---|---|---|
| 资源、时间、区域、设施 | MVU `stat_data` | 模型按白名单规则结算 | 模型投影、庭园地图、目标菜单、顶栏 | 回复与 MVU 复读成功后生效；失败不预扣 |
| 正式事件 | `events.active_event`、`waiting_events`、`completed_key_events` | 模型按事件配置写入 | 模型、设施页、战斗入口 | 同时一个主要事件；完成后清空 active 并写结果 |
| 持续交流 | `interaction.current_session` | 模型 | GAL、模型投影 | 唯一会话 UID；结束时幂等写入 `settled_ids` |
| 有效交流轮数 | `interaction.current_session.effective_rounds` | 模型每次完整 assistant 结算递增 | 模型、r20 事件前置、诊断页 | 新会话从 0 开始；结算后随会话清空 |
| 完整剧情 | SillyTavern 真实消息 | 正常消息事务 | 模型、GAL、原生聊天 | 不删除、不设 `is_hidden=true`；Swipe 以当前分支为准 |
| 战斗结果 | `battle.current` | 本地 battle bridge 仅写入经白名单和范围校验的结果 | 模型、战斗结算流程、诊断页 | 成功消费后写 `battle.settled_ids` 并清空；失败保留恢复入口 |
| 移动锚点线索 | 关键事件结果与 `memory.long_term_notes` | 战后模型结算 | 模型投影、后续事件 | r20 只解锁线索，不创建 `anchors.stable` |
| UI 页面、动画与草稿 | iframe 临时状态 | UI | UI | 不成为第二份正式游戏状态；刷新从 MVU 和消息重建 |

新增 `effective_rounds` 时必须同步：schema、字段台账、类型、模型更新规则、模型投影、测试与旧存档默认值。它只记录当前会话已完成的有效 user/assistant 回合数，不记录全文历史。

## 5. r19 状态与操作设计

### 5.1 温室目标菜单

`greenhouse_plot` 与 `magic_greenhouse` 使用状态驱动操作，不使用一组始终可点的静态按钮：

| 条件 | 可用操作 | 禁用或隐藏逻辑 |
|---|---|---|
| 未完成魔理沙线索 | 查看旧地基 | 建设、清理不可用，并说明需要先调查魔力痕迹 |
| 主屋已修复且灵感为 1 | 调查异常生长／听魔理沙设想／研究旧图纸 | 三种入口共享 `gain_second_inspiration`，只能结算一次 |
| 灵感至少 2、地基已解锁 | 清理旧地基 | 有其他主要工程时禁用 |
| 地基已清理、资源足够 | 搭建基础魔法温室 | 明确显示 4 物资、2 灵感及推进一个时段 |
| 地基已清理、资源不足 | 寻找材料／请求协助／继续调查 | 不形成死局，不允许假装建设成功 |
| 温室已启用 | 使用温室／与在场角色交谈 | 妖花核心活跃时转为事件处理入口 |

所有操作必须生成带稳定 `action_id`、`target_id`、`event_id` 的真实玩家消息。LLM 只能在登记的允许结果中演绎，不能由正文临时创造扣费按钮或跳过前置。

### 5.2 r19 结算矩阵

| 事件 | 成功结算 | 时间 | 资源 | 设施／区域结果 |
|---|---|---:|---:|---|
| `marisa_material_rumor` | 写入允许结果之一 | 不强制 | 无 | 解锁 `greenhouse_plot`；`magic_greenhouse` 进入“可建设”认知状态 |
| `gain_second_inspiration` | 写入三种来源之一 | 不强制 | 灵感 +1，最多到 2 的本事件目标值 | 不重复奖励 |
| `clear_greenhouse_foundation` | `foundation_cleared` 或 `hidden_root_network_found` | +1 时段 | 无 | 区域状态改为“已清理” |
| `build_basic_magic_greenhouse` | `basic_greenhouse_enabled` 或 `enabled_with_instability` | +1 时段 | 物资 -4、灵感 -2 | `state=启用`、`current_form=基础魔法温室`；不稳定结果写入 `active_effects`，不阻断首次使用 |
| `greenhouse_first_use` | `stable_first_growth` 或 `unusual_growth_observed` | 按实际内容，默认不强制 | 无 | 保持启用；异常生长作为妖花事件的正文证据 |

任一事件只有在 assistant 回复完整、允许结果合法且 MVU 复读成功后才扣资源、推进时间或写完成标记。重复结算不得再次扣费或推进。

## 6. r20 会话、事件与战斗设计

### 6.1 持续交流

- 第一次使用温室完成后，提供与魔理沙继续研究或交谈的入口。
- 会话必须使用真实消息，允许建议回复、手动输入、Swipe、停止后续写与显式结束。
- `effective_rounds >= 2` 且会话自然结束后，才能写入 `greenhouse_multiturn_conversation` 完成结果。
- 刷新或切页从 MVU 恢复会话；失败回复、停止生成和被 Swipe 掉的分支不增加正式轮数。

### 6.2 妖花核心触发

只有以下条件同时成立时显示正式处理入口：

- `greenhouse_first_use` 已完成；
- `greenhouse_multiturn_conversation` 已完成；
- 当前没有其他不可让位的主要事件；
- `battle.current` 为空；
- 妖花核心尚未结算。

事件激活后，温室页切换为事件表面，提供“调查核心”“准备符卡战”“采用叙事处理”等受控入口。普通“使用温室”不得绕开事件。

### 6.3 战斗信任边界

1. UI 只加载本地登记的 `greenhouse_flower_core_tutorial_v1`。
2. 战斗引擎只输出 schema 已允许的字段和四种结果。
3. bridge 校验 `config_id`、数值范围、事件前置和 `settlement_id` 唯一性后写入 `battle.current`。
4. UI 随后发送一条真实战斗结算消息，模型只能消费 `battle.current`，不能采用正文伪造的第二份结果。
5. 模型写入剧情结果、`battle.settled_ids`、事件完成标记后清空 `battle.current`。
6. 生成失败时保留待结算结果和重试／原生恢复入口；重复提交不得再次结算。

### 6.4 战斗结果映射

| outcome | 温室结果 | 锚点线索 | 后续 |
|---|---|---|---|
| `clean_win` | `state=启用`，清除妖花不稳定效果 | 完整线索 | 温室正常开放 |
| `narrow_win` | `state=启用`，保留“根系余波待观察”效果 | 完整线索 | 后续可出现短恢复事件 |
| `loss` | `state=异常`，不得永久毁坏或封死 | 残缺但可用的线索 | 提供恢复、求助或重试路线 |
| `narrative` | `state=启用`，保留轻微异常效果 | 完整线索 | 作为无障碍／跳过弹幕的合法完成方式 |

所有结果均结束 M1 妖花核心事件并留下移动锚点线索；r20 不直接创建稳定锚点，不允许精确选择传送目的地。温室失败路线必须可恢复，不能造成资源或剧情死局。

## 7. 维护源变更范围

预计修改：

- `src/ui/target-actions.ts`：温室、魔理沙、妖花事件操作及禁用原因；
- `src/ui/app.ts`：设施事务表面、事件入口、状态反馈和受控战斗启动；
- `src/ui/bridge.ts`：战斗结果校验、暂存、复读和恢复；
- `src/ui/message-transaction.ts`：设施／战斗结算的幂等阶段；
- `src/ui/types.ts`：温室事件、会话轮数和战斗状态类型；
- `src/ui/index.html`、`styles.css`：温室操作、消耗预览、异常与战斗恢复表面；
- `src/schema/02-mvu-schema.js`、`initial-state.json`、`field-ledger.md`：`effective_rounds` 与战斗恢复契约；
- `src/lorebook/events/greenhouse-vertical-slice.json`：补齐机器可判定的前置、结果与状态映射；
- `src/lorebook/variable-update-rules.md`、`model-projection.md`：r19/r20 结算与投影；
- `src/lorebook/routing-plan.json`：确保当前事件配置到达正确模型阶段；
- `tests/ui-contract.test.mjs`：状态链、禁用逻辑、幂等、战斗白名单与回归测试。

仅在实际字段链需要时修改 schema；不得为纯 UI 进度、动画或战斗画面新增持久字段。所有修改从维护源产生，禁止直接编辑 `dist/`。

## 8. 实施顺序

### 阶段 A：契约与测试夹具

1. 将事件配置从自然语言前置补充为可供 UI 判断的稳定条件。
2. 定义 `effective_rounds` 和战斗暂存／消费链。
3. 为完整初始状态、主屋完成态、资源不足态、温室建成态、妖花活跃态和待结算战斗态建立夹具。
4. 先增加共享状态机与幂等逻辑的自动测试。

### 阶段 B：完成 r19 源码里程碑

1. 接入魔理沙线索和第二点灵感。
2. 接入地基清理、建造预览与合法结算。
3. 接入首次使用与状态驱动温室表面。
4. 执行 `npm test`、`npm run check:ui`、`npm run build:ui`。
5. 不等待人工验收，记录结果后继续 r20。

### 阶段 C：完成 r20 集成里程碑

1. 接入持续交流轮数与结算。
2. 接入妖花事件调度和战斗入口。
3. 完成 battle bridge 的验证、暂存、消费和失败恢复。
4. 接入四种结果、温室变化和锚点线索。
5. 执行完整自动门禁与打包 dry-run；未经授权不写正式包。

### 阶段 D：联合验收与后续修复

联合候选完成后按以下批次逐步验收，不要求一次测完：

1. 基线回归：导入、新聊天、开场、灵梦、主屋、长正文；
2. r19 主链：魔理沙、灵感、地基、资源不足、建造、首次使用；
3. r20 主链：多轮交流、妖花事件、四种战斗结果、锚点线索；
4. 生命周期：刷新、Swipe、停止续写、切聊天、切卡、长聊天；
5. 界面：窄屏、触控、键盘、长文本、原生恢复；
6. 失败恢复：生成失败、MVU 复读失败、待结算战斗、重复点击。

验收发现的问题按严重度进入后续新检查点：主流程阻断优先，状态错误其次，体验与美术问题最后。不得覆盖 r19/r20 已生成的检查点目录。

## 9. 自动验证门

### r19 门

- 温室操作只在合法条件下启用；
- 第二点灵感只结算一次；
- 建造不预扣，失败不改变资源和时间；
- 两种建设结果都允许首次使用；
- r18 核心契约测试无回归；
- `npm test`、`npm run check:ui`、`npm run build:ui` 全部通过。

### r20 门

- 少于两轮有效交流不能完成持续交流事件；
- 非妖花事件状态不能启动对应战斗；
- 非法配置、越界数值、重复 settlement ID 被拒绝；
- 四种 outcome 都有可恢复且不死局的状态映射；
- 战斗结算失败时 `battle.current` 可恢复，成功后必定清空；
- 锚点线索存在，但 `anchors.stable` 不被提前创建；
- 全量自动测试、类型检查、UI 构建和 package dry-run 通过。

自动门禁不能替代真实 Luker 验收。

## 10. 完成定义

r19+r20 联合开发只有同时满足以下条件才算“开发完成”：

1. 从 r18 已验收基线可连续走通温室建设、首次使用、多轮交流、妖花事件和战斗结算；
2. schema、初始化、更新规则、投影、UI、结算、清理和测试一致；
3. 所有资源与时间变化均有一次性、幂等、复读成功的证据；
4. 失败、叙事替代和资源不足均不形成死局；
5. r18 的同层壳、真实消息、GAL 长正文和恢复入口没有回归；
6. 自动门禁全部通过；
7. 联合候选、已知风险和待真机项目写入交接文档；
8. 正式打包、导入和旧角色卡／世界书清理已按所有者授权完成；历史聊天目录保留。

- M2 的妖精花园、月见温泉与宴会广场；
- 八名主要角色的全量操作与事件；
