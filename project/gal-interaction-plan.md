# 0.2.0-r15 GAL 单壳交互计划

## 1. 状态与目标

- 决策状态：2026-07-23 已由所有者确认。
- 运行基线：已验收的 `0.2.0-r14`。
- 目标卡类型：MVU 角色卡、真实 SillyTavern 消息链、本地自包含同层 UI 的混合架构。
- 本阶段只制作可验证的垂直切片，不打包、不导入、不覆盖 r14，除非所有者后续明确授权。

r15 将现有“剧情消息列表＋输入框”改为玩家视觉上的单一 GAL 游戏页面。后台仍创建真实 user/assistant 楼层，继续承担模型上下文、Swipe、重新生成、MVU 楼层状态和聊天导出。

## 2. 已确认决策

1. 结束聊天完成一次交互回合，但是否推进时段由实际内容或事件配置决定。
2. 结束聊天发送真实离场消息，并保留一次 LLM 收尾回复与幂等 MVU 结算。
3. GAL 回复播放完后提供 2–4 个建议回应、手动输入和结束聊天。
4. “摸摸头”等直接动作进入完整 GAL 会话，不使用一次性文本弹窗。
5. 一条 assistant 回复最多拆成 6 个演出片段；全部播放完后才能发送下一次回应。
6. r15 不做流式 GAL 解析；等待 assistant 楼层与 MVU 更新完整落盘后再播放。
7. 活跃会话中切换目标时，只允许返回、结束后切换、条件式邀请加入或取消。

## 3. 不变契约

- `stat_data` 是唯一正式游戏状态源。
- 完整剧情历史只存在于 SillyTavern 真实消息。
- 所有正式 user/assistant 消息保持 `is_hidden=false`，游戏模式只隐藏其 DOM。
- 常驻壳不绑定真实消息 0 楼；长聊天、懒加载和消息重建不得移除游戏界面。
- 数据库只在正式结算后归档，不成为第二状态源。
- UI 不执行模型给出的 HTML、脚本、URL、图片路径或未登记操作。
- 切卡、切聊天、脚本卸载、心跳丢失或未捕获错误时恢复原生聊天。

## 4. 状态归属

| 内容 | 唯一归属 | 写入者 | 读取者 | 清理或恢复 |
|---|---|---|---|---|
| 完整剧情与分支 | SillyTavern 消息 | 正常消息事务 | 模型、GAL 投影器、原生恢复页 | 随聊天保存；不由 UI 删除 |
| 正式游戏状态 | MVU `stat_data` | 模型更新协议或已验证事务 | 模型、地图、GAL、设施页 | 按 schema 迁移 |
| 当前主要会话 | `interaction.current_session` | 模型结算协议 | 模型、GAL、目标路由 | 幂等结算后清空 |
| 可用目标操作 | 本地操作注册表＋MVU 派生 | 维护源配置 | 目标菜单 | 每次状态刷新重新计算 |
| 当前页面、片段索引、焦点 | iframe UI 状态 | UI | UI | 刷新时从当前会话和最新楼层重建 |
| 未发送输入草稿 | 按 chatId 隔离的会话草稿 | 玩家/UI | GAL 输入区 | 发送确认后清理；失败时保留 |
| 立绘映射 | 本地素材注册表 | 维护源 | GAL 渲染器 | 缺失时回退默认图 |

GAL 片段、当前立绘、动画帧和点击进度不是正式剧情状态，不写入 MVU。刷新后可重新播放最新有效 assistant 回复；若保留片段进度，只能使用与 `chatId + messageId + swipeId` 绑定的临时 UI 记录。

## 5. 单壳页面模式

常驻壳只允许一个活动实例，内部路由包含：

- `opening`：确定性开场；
- `garden`：庭园地图；
- `target_menu`：角色、区域或设施附近的操作菜单；
- `gal_scene`：角色近景和逐段对话；
- `facility_scene`：检查、施工、改造和设施剧情；
- `battle`：JSON 符卡战；
- `settings_recovery`：诊断与原生聊天恢复。

模式切换不重建宿主壳。Canvas、动画时钟和消息订阅由统一生命周期控制器管理。

## 6. 目标操作注册表

角色、区域和设施使用统一目标键：

```text
target_type: character | area | facility
target_id: stable id
```

每个入口操作至少声明：

- 稳定 `action_id`；
- 玩家可见标签和说明；
- 目标类型与 ID；
- 前置条件、阻断条件与禁用原因；
- 发送给消息事务的确定性行动意图；
- 进入的页面模式；
- 关联角色、设施或事件；
- 是否可能推进时段；
- 重复点击和失败行为。

入口操作由本地配置与 MVU 计算。LLM 可以决定角色如何回应以及允许结果中的哪一种成立，但不能创造新的可执行操作、消费未声明资源或跳过阻断条件。

## 7. GAL 表现模型

assistant 回复在去除 MVU 更新块后必须仍能作为普通文本阅读。可选的表现信封版本为 `scene.v1`，其派生模型包含：

- `beats`：1–6 个演出片段；
- `kind`：`narration`、`speech` 或 `action`；
- `speaker_id`：旁白时可为空，角色时必须为已登记 ID；
- `reaction_id`：白名单反应；
- `pose_id`：白名单姿态；
- `text`：仅作为纯文本渲染；
- `suggested_replies`：0–4 个可选回应，只包含稳定 ID、玩家可见标签和发送意图。

渲染规则：

- 使用 DOM `textContent`，不将模型内容写入 `innerHTML`。
- 未知角色、反应或姿态回退到旁白或默认立绘。
- 表现信封缺失、截断或非法时，用默认立绘显示完整可读正文，不阻断会话。
- 所有片段播放完后才启用建议回应、手动输入和结束聊天。
- 建议回应只是预填或发送真实 user 消息，不直接改写 MVU。

具体序列化边界在实现前写入独立表现协议和模型提示，并通过格式破损夹具验证；不得仅依赖正则猜测任意自然语言。

## 8. 消息与会话状态机

```text
garden
  -> target_menu
  -> opening_session
  -> submitting_user
  -> generating
  -> awaiting_mvu
  -> presenting_beats
  -> awaiting_player
  -> submitting_user（继续）
  -> requesting_end
  -> generating_closure
  -> settling
  -> garden

任意消息阶段失败
  -> recoverable_error
  -> retry | edit_draft | continue_generation | show_native
```

约束：

- 同一事务重复点击不创建第二条 user 消息。
- 事务冻结 `chatId`；切换聊天后旧事务不得写入新聊天。
- 停止生成后保留已经创建的 user 楼层和输入快照。
- assistant 楼层出现不代表 MVU 已经完成；必须按已验证事件时序复读状态。
- Swipe 或重新生成后，以当前 `messageId + swipeId` 重新构建表现和 MVU 投影。
- 0 层游戏壳本身不参与 Swipe。

## 9. 会话结束与时间

“结束聊天”执行：

1. 根据当前会话生成一条可编辑的离场意图；
2. 玩家确认后创建真实 user 楼层；
3. LLM 生成自然收尾、覆盖式短摘要和允许的状态变化；
4. MVU 写入唯一结算 ID，并复读确认；
5. 数据库可选归档；
6. 清空 `interaction.current_session`；
7. 返回庭园。

同一结算 ID 只允许成功一次。完成会话代表完成一个交互回合，但时间推进遵循：

- 简短寒暄、轻微肢体互动通常不推进；
- 长时间交流、共同调查可推进；
- 维修、建设、探索和正式事件按事件配置推进；
- 失败、停止生成、非法回复或未复读成功不得推进。

## 10. 立绘与反应素材

立绘注册表至少包含：

- `asset_id`；
- `character_id`；
- `reaction_id`；
- `pose_id`；
- 可选服装或场景标签；
- 本地自包含素材路径；
- 默认回退资产。

LLM 只选择白名单语义标签，不直接选择文件路径。r15 允许只提供通用占位图或每个样板角色一张默认图。正式素材待消息、Swipe、结算和恢复链通过后批量补充。

建议首批反应集合：`neutral`、`smile`、`annoyed`、`surprised`、`serious`、`shy`、`sad`、`angry`。实现时可按角色缩减，不要求一次补齐。

## 11. 设施事务

设施和区域操作遵循：

```text
读取状态
-> 计算操作与禁用原因
-> 预览消耗、参与角色和时间影响
-> 玩家确认
-> 创建真实行动消息
-> 播放“进行中”循环动画
-> 等待 assistant 与 MVU
-> 校验允许结果和唯一结算
-> 播放结果动画
-> 返回地图或进入后续剧情
```

在正式结算前不得扣除资源、改变设施状态或推进时间。生成失败时进行中动画停止，保留重试、编辑和原生恢复入口。

设施不用重复投入资源的数值等级。按钮使用“维修”“建设”“使用”“调查”“改造”或具体形态名；形态变化必须来自已登记剧情和玩家选择。

## 12. r15 垂直切片

### 角色样板：博丽灵梦

- 地图命中灵梦后显示目标菜单。
- 操作至少包含“对话”“摸摸头”“离开”。
- “对话”和“摸摸头”均进入 GAL。
- 使用占位近景图。
- 支持 1–6 段点击播放、建议回应、手动输入和结束聊天。
- 角色是否接受动作由性格、关系事实和当前剧情决定。

### 区域样板：旧主屋

- 点击 `areas.main_house` 后显示“检查”“维修”“离开”。
- “维修”接入既有 `main_house_repair` 事件。
- 检查前置、物资和主要工程阻断。
- 生成期间显示简易施工循环动画。
- 只接受登记结果 `main_house_enabled` 或 `temporary_shelter_only`。
- 成功结算后按事件配置推进一个时段。

## 13. 维护源变更范围

实施时预计新增或修改：

- `src/ui/index.html`：单壳模式容器、GAL 与设施表面；
- `src/ui/app.ts`：页面路由和目标选择；
- `src/ui/bridge.ts`：消息、Swipe、停止、结算和恢复桥接；
- `src/ui/message-transaction.ts`：GAL 与设施事务状态；
- `src/ui/types.ts`：目标、表现、立绘和会话类型；
- `src/ui/styles.css`：近景、对话框、窄屏和软键盘布局；
- `src/ui/garden-map.ts`：目标命中与操作菜单入口；
- `src/runtime/ui-host-shell.js`：游戏模式下受控隐藏原生输入区，并保持跨卡恢复；
- `src/schema/*`：仅在字段台账确认现有 `current_session` 不足时最小扩展；
- `src/lorebook/*`：表现协议、会话结束和设施允许结果；
- `src/assets/*`：占位立绘与素材注册表；
- `tests/*`：解析、事务、降级和宿主生命周期契约。

所有变更先改维护源再构建，不直接修改 r14 打包产物。

## 14. r15 验收门

### 静态与离线

- 表现信封合法、缺失、截断、未知 ID、超长文本均有夹具。
- 模型文本只以纯文本渲染。
- 操作注册表拒绝未知 action、无前置条件和重复结算。
- 约 320px 宽度、短视口、200% 缩放和移动端键盘可操作。
- 占位素材缺失不白屏。

### 真实 Luker

- 新聊天与 r14 旧聊天均可进入单壳。
- 后台消息真实存在、顺序完整、`is_hidden=false`，默认不向玩家展示。
- GAL 手动输入和建议回应均创建真实 user/assistant 楼层。
- 一条回复按顺序播放，最后才开放输入。
- 停止、失败、重试、继续生成不会重复 user 消息。
- Swipe、重新生成后文本、立绘、建议回应和 MVU 同步切换。
- 刷新、切聊天、长聊天加载后壳仍为单例，并恢复当前会话。
- 结束聊天只结算一次，并按内容正确决定时间推进。
- 旧主屋维修失败不预扣资源，成功后只落入允许结果。
- 显示原生聊天后消息和输入区完整；返回游戏后再次受控隐藏。
- 切换其他角色卡后壳、样式和隐藏类完全清理。

## 15. 本阶段不包含

- 正式批量角色近景图；
- 流式 GAL 文本解析；
- 八名角色全部操作；
- 四设施完整内容；
- 自动运行时生图或任意远程图片；
- 多个主要会话并行；
- r15 打包、导入或发布。

## 16. 实施前工程探针

这些是版本相关的工程验证，不需要所有者再做玩法选择：

1. 确认当前 Luker/Tavern Helper 中 assistant 楼层落盘、MVU 完成和生成结束事件的可靠顺序。
2. 确认左、右 Swipe 与继续生成的精确调用和事件负载。
3. 确认游戏模式隐藏原生输入区后，停止生成与魔法棒功能可由自定义 UI 安全替代或从恢复页进入。
4. 确认长聊天加载更多历史时，宿主只隐藏楼层而不触发壳重挂。
5. 确认旧聊天缺少 `scene.v1` 时的降级显示不会污染下一轮模型上下文。
