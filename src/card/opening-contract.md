# 动态开场事务契约

0 号楼设置页只维护本地草稿：姓名、称谓/代词、外貌简述、庭园名称。预览和修改草稿不得写入 MVU。

确认时：

1. 校验并冻结草稿快照；
2. 生成一条可编辑的真实 user 开场消息；
3. 通过正常发送与生成链提交，不直接替换 `stat_data`；
4. 正文模型描述玩家抵达荒废庭园并开始自由观察；
5. 变量模型在同一正常回复中写入玩家资料、庭园名，将 `meta.initialized` 与 `meta.opening_committed` 置为 true；
6. 若生成失败，保持 opening 未提交并允许安全重试；禁止产生重复 user 消息。

若真实开场消息已经存在，但 assistant 回复缺少可解析的变量更新：

1. 开场页进入“状态待修复”，不得重新展示成尚未提交；
2. 有回复时允许重新生成最新回复；无回复时只重新触发生成；
3. assistant 已完整回复后，显示“确认资料并进入庭院”；点击时从带当前聊天事务标记的原始 user 消息确定性提取姓名、称谓、外貌和庭园名；
4. 若消息楼层缺少完整 `stat_data`，先从项目初始状态补齐，再仅覆盖开场字段、庭守钥状态和两个 meta 标志；
5. 确定性写入必须落在开场后的最新 assistant 楼层，并在切换界面前复读验证；
6. 仍允许发送带事务标记的受限修复消息，但它只是次要手段；
7. 所有恢复操作必须幂等，不得推进时间、资源、关系或事件；
8. 开场页始终提供“显示原生聊天”，不能把玩家锁在同层界面。

快速开始优先读取 Persona。缺失字段使用“中性称谓”和“无名庭园”，外貌可为空。叙事人称由玩家预设和实际输入决定。

验收必须覆盖：新聊天、刷新后重开、失败重试、重复点击、Swipe 分支，以及实际消息楼层中的持久状态。
