# 确定性开场事务契约

0 号楼设置页只维护当前聊天的本地草稿：姓名、称谓/代词、外貌简述、庭园名称。预览和修改草稿不得写入 MVU。

新聊天确认时：

1. 校验并冻结草稿与聊天 ID；
2. 等待目标 MagVarUpdate 的 `Mvu` 全局就绪；
3. 选择当前聊天已有的首个 assistant 楼层作为精确写入目标；存在普通 user 消息时拒绝走新开场路径；
4. 从 `initial-state.json` 补齐共享默认状态，同时保留目标楼层已有的未知字段和非开场正式状态；
5. 只覆盖玩家姓名、称谓、外貌、庭园名、庭守钥状态及 `meta.initialized`、`meta.opening_committed`；
6. 使用 `Mvu.replaceMvuData` 写入同一楼层并立即复读；
7. 只有复读字段完全一致后才进入庭院并允许数据库做可选归档；
8. 整个确认过程不创建 user 消息、不触发生成、不依赖 LLM 变量块。

初始化必须幂等。相同资料重复提交直接视为成功；若当前聊天已用另一组资料完成开场，则拒绝静默覆盖。写入失败时保留草稿和开场页，不推进时间、资源、关系或事件，并始终允许返回原生聊天。

玩家进入庭院后的第一次真实行动才通过正常 user 消息与生成链调用 LLM。正文模型读取已经持久化的 MVU 投影，从玩家刚抵达荒废庭园后的自由观察开始；变量模型不得再次初始化玩家资料或重复发放庭守钥。

旧版聊天若已经存在带 `gensokyo_opening` 标记的 user 消息，继续使用兼容恢复路径：只从该原始消息读取开场字段，写入其后最新 assistant 楼层并复读验证。受限 LLM 修复只作为旧聊天的末级手段，不能重新成为新开场主路径。

快速开始优先读取 Persona。缺失字段使用“中性称谓”和“无名庭园”，外貌可为空。叙事人称由玩家预设和实际输入决定。

验收必须覆盖：新聊天零生成初始化、复读持久化、刷新后重开、重复点击、聊天切换、写入失败保留草稿、旧聊天兼容恢复，以及首次真实行动的正常生成。
